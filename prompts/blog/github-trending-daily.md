# 任务

你要为 Astro Paper 博客生成一篇中文《GitHub 项目日报》的项目列表。

- 任务：{task}
- 日期：{date}

# 输入证据

下面是脚本抓取的 GitHub Trending daily 榜单、仓库元数据、README 正文摘录与数据边界。只能依据这些证据写作；不要补不存在的项目能力、商业背景、融资、热度原因或第三方评价。

```md
{source_text}
```

# 输出要求

只输出可归档的 Markdown 正文，不要输出 frontmatter，不要输出代码围栏。

文章只需要列出 GitHub Trending 项目，不要写总结、趋势观察或数据边界章节。

# 项目列表格式

- 必须按输入证据中的排名顺序列出每个项目；不要只精选部分项目。
- 每个项目必须用 `## 1. [owner/repo](https://github.com/owner/repo)` 这种二级标题格式，序号使用输入证据里的排名。
- 每个项目标题下必须按固定顺序输出这些字段：
  - `- 项目总结：...`
  - `- 技术栈：...`
  - `- 使用场景：...`
  - `- Stars：...`
  - `- Forks：...`
  - `- 今日新增 Stars：...`
- `项目总结` 是 AI 基于描述与 README 摘录归纳出的 1 句话，说明项目是什么、解决什么问题；不要直接摘抄 README。
- `技术栈` 必须优先使用输入证据里的语言、框架、工具链、模型、运行时、平台等；证据不足时写“未明确”，不要脑补。
- `使用场景` 是 AI 基于描述与 README 摘录归纳出的 1 句话，说明适合谁、用在哪类任务。
- `Stars`、`Forks`、`今日新增 Stars` 的数值必须来自输入证据，不要换算、不要脑补、不要省略。

# 写作规则

- 中文、清晰、克制，像 GitHub Trending 历史榜单记录，不写分析文章。
- 每个项目只记录可由证据支持的总结、技术栈和使用场景，不要扩展成趋势解读。
- 不要把 README 自述当成已验证事实；使用“项目用于”“适合”“可用于”“README 显示”这类克制表述。
- 不要编造 README 没有的功能、框架、公司、协议、价格、路线图。

# 禁止内容

- 不要输出 `## 总结`、`## 今日项目精选`、`## 趋势观察`、`## 数据边界`。
- 不要输出 `描述`、`语言`、`README 摘要` 这些字段名；它们只是输入证据，不是最终页面字段。
- 不要写“值得关注”“不容错过”“革命性”“颠覆”“赋能”等营销腔。
- 不要生成投资建议、商业判断、融资猜测或项目安全背书；不要出现“投资建议”四字，即使是否定或免责声明。
- 不要输出“待补充”“示例”“无法判断”“本文将”等占位或元叙事。

# 输出前自检

输出前在心里检查，但不要把自检过程写出来：

- 是否严格只有 Markdown 正文。
- 是否没有总结、趋势观察、数据边界章节。
- 是否按排名列出每个输入项目，而不是精选项目。
- 每个项目是否都有项目总结、技术栈、使用场景、Stars、Forks、今日新增 Stars。
- 是否没有 README 摘要、描述、语言这些最终页面不需要的字段名。
- 是否没有重复项目。
- 是否没有营销腔和证据外脑补。
