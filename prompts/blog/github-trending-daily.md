# 任务

你要为 Astro Paper 博客生成一篇中文《GitHub 项目日报》。

- 任务：{task}
- 日期：{date}

# 输入证据

下面是脚本抓取的 GitHub Trending daily 榜单、仓库元数据、README 摘录与数据边界。只能依据这些证据写作；不要补不存在的项目能力、商业背景、融资、热度原因或第三方评价。

```md
{source_text}
```

# 输出要求

只输出可归档的 Markdown 正文，不要输出 frontmatter，不要输出代码围栏。

文章必须包含以下结构：

- `## 总结`
- `## 今日项目精选`
- `## 趋势观察`
- `## 数据边界`

# 写作规则

- 中文、清晰、克制，像给技术读者看的开源项目日报。
- `## 总结` 用 2-3 段概括今日榜单的主要方向、语言/技术分布和项目类型，不要只复述项目列表。
- `## 今日项目精选` 从证据中选择 5-8 个最有代表性的项目；每个项目必须用 `### [owner/repo](https://github.com/owner/repo)` 作为三级标题。
- 每个精选项目写 1 段即可，必须包含：它解决什么问题、证据中的技术栈/语言/README 线索、适用场景或读者价值。
- `## 趋势观察` 提炼 2-4 条趋势，必须来自多个项目的共同特征，例如 AI agent、开发者工具、基础设施、设计系统、移动端、自动化、安全等。
- `## 数据边界` 必须说明 GitHub Trending 只是 GitHub 页面榜单，README 是项目自述，today stars 可能随页面刷新变化。

# 禁止内容

- 不要写“值得关注”“不容错过”“革命性”“颠覆”“赋能”等营销腔。
- 不要生成投资建议、商业判断、融资猜测或项目安全背书。
- 不要把 README 自述当成已验证事实；使用“项目自述”“README 显示”“榜单元数据表明”等表述。
- 不要编造 README 没有的功能、框架、公司、协议、价格、路线图。
- 不要输出“待补充”“示例”“无法判断”“本文将”等占位或元叙事。

# 输出前自检

输出前在心里检查，但不要把自检过程写出来：

- 是否严格只有 Markdown 正文。
- 是否包含四个固定二级章节。
- 是否至少包含 5 个带 GitHub 链接的项目标题。
- 是否没有重复项目。
- 是否没有营销腔和证据外脑补。
