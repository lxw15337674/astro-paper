# 任务

你要根据脚本抓取的 Hacker News 证据，生成可交给 Astro archive 层归档的中文 HN Top 10 Markdown。

- 任务：{task}
- 日期：{date}

# 输入证据

下面是脚本抓取并整理的结构化证据。只能依据这些信息写作；不允许编造不存在的原文细节、作者、数据或评论观点。

```md
{source_text}
```

# 输出要求

只输出一个合法 JSON 对象，不要输出 Markdown、frontmatter、代码围栏或任何解释。

热度、主题、原文链接、HN 讨论链接等事实字段由脚本从证据里填充，你不需要输出它们；
你只需为每个条目产出「中文标题 + 内容总结 + 评论总结」。archive 层会把你的 JSON 与
脚本事实按 `rank` 合并，生成最终博客文章。

JSON 结构必须严格如下（`items` 覆盖输入证据里的全部条目，`rank` 与证据编号一一对应）：

```json
{
  "description": "今日最热 2-3 个话题的一句话概括，40字以内",
  "items": [
    {
      "rank": 1,
      "title_zh": "示例中文标题",
      "content_summary": "针对该条目的内容总结……",
      "comment_summary": "针对该条目的评论总结……"
    }
  ]
}
```

- `rank` 必须与输入证据里对应条目的编号一致，且覆盖全部条目，不遗漏、不新增。
- 不要输出「今日 HackerNews 热门文章 Top 10」这类总览条目；证据里的总览行不算条目。

# 标题翻译要求

- 每个条目的主标题必须翻译成自然中文，写入 `title_zh`，不能直接使用英文原题。
- 中文标题要忠实表达原题含义，不要标题党化、不要扩写成观点句。
- 专有名词、产品名、公司名、协议名、编程语言名可以保留英文，例如 Rust、SQLite、OpenAI、Postgres。
- 你不需要也不应该在 JSON 里输出任何链接；链接由脚本按 `rank` 回填。

# 质量边界

- 每个条目的“内容总结”和“评论总结”都必须是自然中文完整段落，不要写空洞模板。
- “内容总结”必须优先依据该条目的“原文正文”写作；不能只根据标题、分数或链接猜测。
- “评论总结”必须优先依据“HN 评论样本”写作；如果评论样本不足，只能保守说明讨论热度和可确认的分歧边界。
- 不要写“信息不足”“待补充”“原文页面提取失败”等占位话术。
- 如果某条确实缺少原文正文，只保守说明从标题、来源和 HN 讨论可确认的内容。
- 不要使用英文长段落，除非是标题或专有名词。
- 不要把评论观点说成事实结论。
