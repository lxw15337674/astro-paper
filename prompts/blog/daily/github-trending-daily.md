# 任务

你要为 Astro Paper 博客生成一篇中文《GitHub 项目日报》的项目列表。

- 任务：{task}
- 日期：{date}

# 输入证据

下面是脚本抓取的 GitHub Trending daily 榜单、仓库元数据、README 正文与数据边界。只能依据这些证据写作；不要补不存在的项目能力、商业背景、融资、热度原因或第三方评价。

```md
{source_text}
```

# 输出要求

只输出一个合法 JSON 对象，不要输出 Markdown、frontmatter、代码围栏或任何解释。

链接（owner/repo、GitHub 地址）与 Stars、Forks、今日新增 Stars 等数值由脚本从证据回填，你不要输出它们。
你只需为每个项目产出「项目总结、技术栈、使用场景」三个语义字段。archive 层会按 `rank` 把你的 JSON
与脚本事实合并，生成最终博客文章。

JSON 结构必须严格如下（`items` 覆盖证据里的全部项目，`rank` 与证据排名一一对应，不遗漏、不新增）：

```json
{
  "description": "今日榜单主要方向与亮点项目，50字以内",
  "items": [
    {
      "rank": 1,
      "project_summary": "1 句话说明项目是什么、解决什么问题",
      "tech_stack": "语言 / 框架 / 运行时 / 平台",
      "use_case": "1 句话说明适合谁、用在哪类任务"
    }
  ]
}
```

- `project_summary` 是基于描述与 README 正文归纳出的 1 句话，不要直接摘抄 README。
- `tech_stack` 必须优先使用证据里的语言、框架、工具链、模型、运行时、平台等；证据不足时写“未明确”，不要脑补。
- `use_case` 是基于描述与 README 正文归纳出的 1 句话，说明适合谁、用在哪类任务。
- 不要在 JSON 里输出任何链接或 Stars/Forks 数值；它们由脚本按 `rank` 回填。

# 写作规则

- 三个语义字段用中文、清晰、克制，像 GitHub Trending 历史榜单记录，不写分析文章。
- 每个项目只记录可由证据支持的总结、技术栈和使用场景，不要扩展成趋势解读。
- 不要把 README 自述当成已验证事实；使用“项目用于”“适合”“可用于”“README 显示”这类克制表述。
- 不要编造 README 没有的功能、框架、公司、协议、价格、路线图。

# 禁止内容

- 不要写“值得关注”“不容错过”“革命性”“颠覆”“赋能”等营销腔。
- 不要生成投资建议、商业判断、融资猜测或项目安全背书；不要出现“投资建议”四字，即使是否定或免责声明。
- 不要输出“待补充”“示例”“无法判断”“本文将”等占位或元叙事。

# 输出前自检

输出前在心里检查，但不要把自检过程写出来：

- 是否只输出一个合法 JSON 对象，没有 Markdown 或代码围栏。
- `items` 是否按排名覆盖每个输入项目，而不是精选项目，`rank` 是否与证据一致。
- 每个项目是否都有 project_summary、tech_stack、use_case 三个非空字段。
- 是否没有在 JSON 里塞链接或 Stars/Forks 数值。
- 是否没有营销腔和证据外脑补。
