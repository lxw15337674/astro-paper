# 核心任务

`{task}`

将本期播客的完整音频内容，转化为一篇可归档、高信息密度、具备知识沉淀价值的中文长文博客笔记。

内容唯一权威来源：本期提供的完整音频；下方元数据仅用于填充「基本信息」模块，不得作为观点论据使用，严禁基于元数据扩写任何观点与内容。

# 本期素材

- 核心内容源：本期播客完整音频（所有观点、论据、案例、数据均必须出自音频原文）
- 基础元数据（仅用于填写基本信息字段）：

```md
{source_text}
```

- 发布日期：`{date}`

# 输出结构与写作要求

严格遵循以下结构顺序输出，第一行必须输出 `DESCRIPTION: <本期核心主题摘要，40字以内>`，第二行为空行，第三行起输出本期播客的二级中文标题，标题前禁止放置任何总览、引言、清单、说明类文字。所有模块标题统一使用三级标题格式。

## 一、必选核心模块

以下模块必须完整输出。

### 1. 本期播客中文标题（`##` 级标题）

- 翻译与命名规则：忠实还原本期核心主题，中文表达自然流畅，禁止标题党化，禁止新增音频未提及的观点与噱头。
- 专有名词规则：产品名、公司名、人名、模型名、节目原名可保留英文，如 OpenAI、Claude、NVIDIA、Latent Space、The Daily，无需强行翻译。
- 注意：原始英文节目名仅出现在「基本信息」模块，本标题为本期内容主题的中文概括。

### 2. 核心主题

用 1-2 句话精准概括本期播客的核心讨论议题与问题意识，明确本期内容围绕什么核心问题展开、最终要回应什么核心疑问。

### 3. 基本信息

以无序列表固定呈现以下字段：

- 节目：播客节目官方原名
- 嘉宾：音频或元数据中明确提及的嘉宾姓名及身份；无明确信息时填写「未标明」，禁止臆测嘉宾身份与背景。
- 日期：本期播客正式发布日期
- 来源：播客出品方 / 首发平台
- 链接：本期播客官方访问链接

### 4. 核心观点

提炼本期播客最终传递的核心立场与本质判断，而非话题清单。要求：

- 直击节目核心结论，明确区分「节目共识性观点」与「嘉宾个人经验判断」，不得将个体经验包装为普适行业结论。
- 控制在 2-3 条核心结论，每条为完整的判断句，具备明确立场与指向性。
- 严禁泛泛表述（例如“本期讨论了 AI 行业发展趋势”为无效内容，必须写明具体趋势判断）。

### 5. Highlights 核心亮点

- 输出 1-5 条精华结论，每条均为可被音频内容直接佐证的完整判断。
- 禁止使用“聊了 XX”“介绍了 XX”这类仅描述话题、无信息增量的表述。

### 6. 长文笔记

这是全文最核心的知识沉淀模块。禁止按时间线流水账复述访谈，禁止压缩为话题列表。要求：

- 围绕 3-6 个高价值核心议题重组内容，每个议题设置独立的四级小标题，逻辑分层清晰，支持快速扫读。
- 每个议题统一遵循「观点结论 -> 支撑论据 / 案例 -> 底层逻辑 / 运行机制 -> 行业影响 / 实践启发」的四层结构展开。
- 必须讲清每个观点的具体内涵、提出依据、支撑案例、背后逻辑，以及对对应受众的参考价值。
- 完整覆盖以下维度：本期核心问题、行业背景语境、核心判断、论证依据、相关案例 / 产品 / 技术 / 人物细节、争议与风险、对从业者的启发。
- 信息密度标准：读者无需收听原音频，即可完整理解本期全部核心内容与完整逻辑链条。

# 通用约束

- 只输出可归档的 Markdown 正文，不要输出 frontmatter，不要输出解释性文字，不要输出代码围栏。
- 所有观点、论据、案例、数据，必须能够从本期完整音频中找到依据。
- 元数据只用于填写「基本信息」模块，不能作为正文观点或论据来源。
- 不得编造嘉宾身份、公司背景、技术细节、行业结论、时间戳、链接或任何未在音频和元数据中明确出现的事实。
- 禁止使用空泛总结、套话、标题党表达，以及任何“本文将”“本期主要聊了”等元叙述性文字。
