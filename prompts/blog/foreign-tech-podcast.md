你要把上游抓取/整理的海外科技播客证据写成中文长文博客笔记。

- 任务：{task}
- 日期：{date}

```md
{source_text}
```

# 输出要求

只输出可归档的 Markdown 正文，不要输出 frontmatter，不要输出代码围栏。

文章结构要求：

1. 第一行固定输出：`《今日国外热门科技访谈播客》`
2. 第一行之后直接进入每期播客的 `## episode title` 小节。
3. 不要输出 `## 今日总览` 或 `## 今日播客清单`，也不要在正文开头额外写总览段或播客清单。
4. 每个 episode 小节必须按以下顺序包含：
   - `### 中文主题`
   - `### 基本信息`
     - `- **节目**：...`
     - `- **嘉宾**：...`，输入没有明确嘉宾时写“未标明”，不要猜。
     - `- **日期**：...`
     - `- **来源**：...`
     - `- **链接**：...`
   - `### 一句话总结`
   - `### Highlights`
     - 3-5 条 bullet，每条必须是可由输入简介支持的技术、产品、职业或产业判断。
   - `### 长文笔记`
     - 至少 3 段，每段围绕一个可复盘观点展开。

不要改变上述结构、标题层级、播客条目顺序或整体版式；不要新增引言、观点对照、读者可以带走什么等额外大结构。优化重点只在每个播客条目内部，把内容讲解写得更厚。

# 质量要求

- 必须是长文中文笔记，不是短消息摘要；每个播客条目都要写到“不听播客也能理解本期具体内容”的程度。
- 重点写访谈里的技术、产品、职业、产业判断和可复盘观点；不要为了简短压缩信息，不限制单个条目的展开长度。
- 每个播客条目的 `### 长文笔记` 应尽量讲清：这期主要问题、问题背景、核心观点、观点理由、例子/事件/产品/人物/公司/技术背景、争议/风险/不确定性、对开发者/创业者/产品/AI 行业或普通读者的启发。
- 每个播客条目必须主要依据该条的 `#### Transcript excerpt` 写作；标题、链接、图片、嘉宾、show notes 只能做元数据补充，不能单独支撑观点展开。
- 凡是出现一个观点，都要解释它具体指什么、为什么这么说、有什么例子或影响；禁止只写“本期讨论了 AI、创业、技术趋势”等泛泛概括。
- 如果某一期播客最终只能让读者知道“大概主题”，但不能知道“里面具体讲了哪些观点、理由和背景”，则视为不合格，必须继续展开该条目。
- 可以做分析和归纳，但只能基于输入证据；不要编造 transcript、嘉宾、链接、来源、节目名、发布日期或未提供的事实。
- 输入源阶段应已过滤掉无 transcript 条目；如果仍看到某条 `#### Transcript excerpt` 为空、缺失或写着未提供 transcript，不要为该条生成小节。
- 不要出现“待补充”“示例”“信息不足”“无法判断”“本文将”等占位或元叙述文字。
- 不要重复输出同一播客的总结内容；`### 一句话总结`、`### Highlights` 和 `### 长文笔记` 必须各自承担不同功能，不能复制粘贴同一句话或同一段观点。
- 不要生成投资建议、产品购买建议或夸张标题。
